# IaCCreator

A visual AWS architecture diagram editor that generates production-ready Terraform projects. Two integrated parts communicating over a REST API with cookie-based anonymous sessions:

- **`app/`** — FastAPI backend (Python 3.12+) that turns a diagram description into a modular Terraform file tree.
- **`frontend/`** — Next.js 16 + React 19 diagram editor (Canvas + DOM hybrid rendering).

See `README.md` for the full feature list and generated output structure. See `docs/*.md` for deep-dives per layer — read the relevant one before making non-trivial changes rather than re-deriving behavior from scratch:

| File | Covers |
|---|---|
| `docs/backend-api.md` | API endpoint reference |
| `docs/backend-generators.md` | Terraform/HCL generator internals (one file per AWS service in `app/generators/`) |

## Commands

### Backend (run from repo root)

Always use `uv` for Python — never `pip`/`python` directly.

```bash
uv sync                                    # install/sync dependencies
uv run uvicorn app.main:app --reload       # dev server, http://localhost:8000
uv run pytest -n auto                      # run all tests in parallel (pytest-xdist)
uv run pytest tests/test_unit.py -k some_name  # run a single test
uv run ruff check .                        # lint
uv run ruff format .                       # format
```

### Frontend (run from `frontend/`)

Always use `pnpm` for the frontend — never `npm`/`npx`/`yarn`. When invoking pnpm from outside the `frontend/` directory (e.g., from repo root), use `npx pnpm` instead of standalone `pnpm`.

```bash
npx pnpm install    # from repo root
pnpm install        # from frontend/ directory

pnpm dev         # dev server, http://localhost:3000
pnpm build
pnpm lint         # eslint
pnpm vitest run   # run all tests
pnpm vitest run path/to/file.test.ts   # run a single test file
```

### Docker (both services together)

```bash
docker compose up --build
```

## Architecture notes

- Backend request flow: `routers/` → `services/ir_builder.py` (validates + builds `ProjectIR`) → `services/code_generator.py` → `services/connection_processor.py` (thin facade, dispatches to `services/connection_handlers/`) → `services/file_tree_assembler.py` → generator classes in `generators/` (one per AWS service) emit HCL via `generators/hcl_renderer.py`. Connection logic lives in `services/connection_handlers/` (one handler file per connection type); the registry dispatches `(source_service, target_service)` pairs to handler instances.
- Anonymous sessions are cookie-based (30-day expiry), handled by `middleware/session_middleware.py`; persistence uses the repository pattern (`persistence/`) with TinyDB locally and DynamoDB in production, selected via `PERSISTENCE_BACKEND` env var.
- Adding a new AWS service generally means: add/extend a Pydantic model in `models/input_models/`, add a generator in `generators/`, register it in `generators/registry.py`, and if it connects to other services, add a handler in `services/connection_handlers/` and register it in `services/connection_handlers/registry.py`.
- Frontend state lives in Zustand stores under `src/store/`; canvas rendering is a hybrid of an HTML5 Canvas (grid/connectors) and DOM overlay (service nodes, panels).
- `.kiro/specs/<feature>/` holds spec-driven design docs (`requirements.md`, `design.md`, `tasks.md`) for larger features — check there for design rationale before assuming intent from code alone.

## Conventions

- Generated HCL must use consistent two-space indentation and Terraform resource references instead of hardcoded values (see `docs/backend-generators.md`).
- Python: ruff-enforced (`pyproject.toml`); domain errors use custom exceptions in `app/exceptions.py`, not bare `HTTPException`, inside services — routers translate them at the boundary.
- Tests are property-based where it matters (Hypothesis on the backend, fast-check on the frontend) — prefer extending existing strategies in `tests/conftest.py` / `frontend/__tests__/properties/arbitraries.ts` over writing narrow example-based tests for generator/serialization logic.
- Commit messages: single-line only. Multi-line `-m` messages break in the maintainer's fish shell. Do not add bullet points or line breaks inside a commit message.
- Frontend `AGENTS.md`/`CLAUDE.md` under `frontend/` were generated by Next.js itself (framework-version warning) — leave as-is, it is not project-specific guidance.
- Never invoke `pip`, bare `python`, or a global interpreter for this project — always go through `uv` (`uv sync`, `uv run ...`, `uv add <pkg>`) so dependencies stay pinned to `uv.lock`.
- Frontend package manager is `pnpm` (pinned via `packageManager` in `frontend/package.json`); `frontend/pnpm-lock.yaml` is the committed lockfile, do not regenerate `package-lock.json`. `frontend/pnpm-workspace.yaml` holds `allowBuilds` for `sharp`/`unrs-resolver` so `pnpm install` doesn't hang on an interactive build-script prompt in Docker/CI.
- `build/Dockerfile.backend` and `build/Dockerfile.frontend` use BuildKit cache mounts (`RUN --mount=type=cache,...`) for the `uv`/`pnpm` package caches — requires BuildKit (default for `docker buildx build` and `docker compose` v2).
- `docker-compose.override.yml` is auto-loaded by `docker compose` alongside `docker-compose.yml` and switches both services to their `dev` build stage with source bind-mounts for hot reload (`uvicorn --reload` for the backend, `pnpm dev` for the frontend). To run the plain production-like build without hot reload, pass `docker compose -f docker-compose.yml up --build` explicitly.
- Tests run in parallel by default (`pytest-xdist`, `-n auto`). Every fixture that builds a `TestClient` from the real `app.main.app` (`tests/test_cors_and_wiring.py`, `tests/test_variable_schemas_endpoint.py`, `tests/test_properties.py`) must, on an isolated `tmp_path`-backed repo: (1) monkeypatch `app.persistence.factory.get_repository`, (2) `importlib.reload(app.main)`, **and** (3) override `app.routers.diagrams.get_repo` via `app.dependency_overrides`. Step 3 is required in addition to 1+2: `diagrams.py` does `from app.persistence.factory import get_repository`, binding its own name at *its own* import time, which can predate the patch depending on test collection order — skip it and diagram-router requests silently fall through to the real `data/db.json`, where concurrent xdist workers corrupt the file (`json.JSONDecodeError`) for every subsequent run until it's deleted. Any new test hitting the real app must follow the full three-step pattern.

## Environment quirks

- The maintainer's interactive shell is **fish**, not bash. This doesn't affect the agent's own `bash` tool, but if you're producing copy-paste commands *for the user* (in chat, docs, or scripts meant to be pasted into their terminal), avoid bash-only syntax (`&&`, `||`, `$(...)`, inline `VAR=val cmd`) or wrap it as `bash -c "..."`.
- When working with Terraform/AWS provider details, prefer an available Terraform MCP server (registry/provider lookups) over relying on training data — provider resource names and arguments change.
